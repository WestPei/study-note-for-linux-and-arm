## 1.2 使用`.iso`镜像创建虚拟机模板

由于服务器无法连接网络，官方的iso镜像没有安装任何软件包，都需要联网下载，所以在个人PC上先创建好模板，再导入服务器。
尝试在个人PC上使用ubuntu 20.04LTS的iso镜像在vmware中新建虚拟机，配置好环境后生成镜像模板。这里遇到了很多坑...
首先是VMWare如果直接选择用iso镜像构建虚拟机，由于iso镜像的设置，会直接采用简易安装的方式，跳过很多自定义步骤（比如语言设置，磁盘分区挂载等），这在安装好系统后再更改会很麻烦，也很容易出问题。
因此选择先构建一个空的虚拟机，为虚拟机添加两个磁盘，一个50g的系统盘，另一个100g的作为挂载在`/home`下的盘，方便以后为构建Yocto工程进行扩容。 然后手动为CDROM选择我们需要用到的iso镜像。这时候启动虚拟机就不会直接简易安装了，而是可以自定义安装选项。
通过这种方式来分配磁盘和分区，能够实现上述结构。

为其安装vim，net-tools等必要的工具后就可以将其打包为qcow2镜像作为模板了。
我们需要将对应的两个vmware镜像文件.vmdk拷贝到一个Ubuntu中（可以是一个虚拟机）
然后利用qemu-img工具将其转换成.qcow2的镜像。
首先是安装工具：

```bash
sudo apt update && sudo apt install –y qemu-utils
```


然后就是转换：

```bash
qemu-img convert –f vmdk –O qcow2 input.vmdk root.qcow2
qemu-img convert –f vmdk –O qcow2 input-disk.vmdk home.qcow2
```

转换完成后可以验证其完整性：
```qemu-img info root.qcow2 和 qemu-img info home.qcow2```

然后将这两个镜像文件导入服务器，作为模板来构建虚拟机。

### 1.2.1 离线安装软件包

由于服务器没有连接互联网，无法直接利用`sudo apt install`命令来下载软件包，因此我们需要一个联网的ubuntu虚拟机来下载软件压缩包`.deb`，然后移动到服务器虚拟机上再进行安装。

这里我们以ubuntu20.04LTS为例，两台虚拟机版本一致。**统一称呼连接互联网的虚拟机为`虚拟机0`，未连接互联网的虚拟机为`虚拟机1`。**

在`虚拟机0`上为要安装的软件包创建单独的目录，方便管理：

```bash
mkdir -p ~/Download/stress-ng
cd ~/Download/stress-ng
```

在下载压缩包前我们还需要一个用于获取软件的所有**依赖包信息**的工具。

```bash
sudo apt update
sudo apt install -y apt-rdepends
```

`apt-rdepends`就是我们需要的工具。安装完成后我们需要得到对应软件包的依赖包列表，这里以`stress-ng`软件为例：

```bash
# 生成stress-ng所有依赖的列表（包括间接依赖）
apt-rdepends stress-ng | grep -v "^ " | grep -v "debconf-2.0" | awk '{print $1}' | sort -u > dependencies.txt
```

执行完成后会在当前目录下生成一个`dependencies.txt`文件。里面记录了所有需要的依赖包。然后就可以在终端执行：

```bash
# 从依赖列表中下载所有包（包括stress-ng本身）
xargs -a dependencies.txt sudo apt-get download
```

上述就是下载命令，会有一些提示信息，完成后在目录下就会出现一些`.deb`格式的文件，这些就是下载好的包文件。

到目前为止我们完成了目标软件及其依赖包的下载。现在就需要通过各种手段将这些文件从`虚拟机0`移动到`虚拟机1`中。这里我使用了移动硬盘+MobaXterm来转移。

我将文件转移到了`虚拟机1`的`~/packages/stress-ng/`目录下。进入该目录，就可以离线安装软件包了：

```bash
sudo dpkg -i *.deb
```

在执行过程中可能会出现“依赖关系问题-仍未被配置”的提示，这可能是因为`dpkg`安装顺序没有处理好依赖关系，但是因为我们完整下载了所有的依赖包并且在同一个目录下，因此这个提示会被自动解决。

安装完成后可以验证一下安装结果：

```bash
stress-ng --version
```

## 1.3 虚拟机磁盘扩容

目前服务器上的虚拟机主要是CentOS和Ubuntu，两种系统的磁盘扩容方式分别给出。

### 1.3.1 Ubuntu虚拟机扩容

注意先关闭虚拟机！该操作可以直接在管理员主机操作，也就是在`192.168.1.6`上来为挂载在`/home`下的磁盘文件镜像ubuntuvmdisk.qcow2文件进行扩充操作，会使用到`qemu-img`命令。
首先找到我们需要扩容的文件，比如这里我们的文件位置为

> /home/public/virt-manager/peijd/ubuntuvmdisk-peijd.qcow2

关闭对应的虚拟机，然后正式开始扩容操作。
可以直接设置绝对大小

```bash
sudo qemu-img resize /home/public/virt-manager/peijd/ubuntuvmdisk-peijd.qcow2  200G
```
也可以增量扩容

```bash
sudo qemu-img resize /home/public/virt-manager/peijd/ubuntuvmdisk-peijd.qcow2 +50G
```

主机方面的扩容就完成了。
然后是虚拟机下对磁盘进行扩容
可以通过`lsblk`来查看磁盘情况，可以发现挂载在`/home`目录下的是vdb，只分了一个区vdb1。
然后我们使用`fdisk`来进行扩容。
首先是切换单用户模式或维护模式

```bash
sudo systemctl rescue
```

或者强制卸载`/home` （其实这里我没有成功卸载）

```bash
sudo umount /home
```

我认为很关键的一步：**使用`fdisk`删除并重建分区**（数据不会丢失，但是起始扇区必须相同

```bash
sudo fdisk /dev/vdb
```

	在fdisk中
	输入 ‘p’可以查看当前分区表，记录一下起始扇区（Start项）这里是2048
	输入 ‘d’用来删除分区 这里只有一个分区vbd1，所以会删除他
	输入 ‘n’用来创建新的分区
	选择 ‘p’也就是新建分区是主分区
	选择 ‘1’为分区号
	注意起始扇区值与原来的保持一致，这里是2048
	结束扇区默认就行，默认为最大。
	输入 ‘w’写入并退出

`fdsik`的操作完成后

```bash
sudo reboot # 重启系统
```

用`lsblk`检查虚拟机的磁盘分区大小，发现vbd1大小变成了200G，说明扩容成功。

```bash
df –h /home # 检查文件系统大小
```

发现文件系统大小还没有扩展，仅仅是虚拟磁盘扩容了。
**`lsblk`用于查看分区大小（物理层面），而`df –h`是查看文件系统大小（逻辑层面）!**
现在用最简单的方法 在线扩展XFS文件系统（这里可以通过`df –T`查看挂载的文件系统类型
直接扩展XFS文件系统

```bash
sudo xfs_growfs /home
```

或者指定设备 

```bash
sudo xfs_growfs /dev/vdb1
```

最后验证结果

```bash
df –h /home
```

一般这样就完成了xfs文件系统的扩容，如果挂载在`/home`下的磁盘使用的其他文件系统，可能命令有所不同。



如果是ext4文件系统，则可以使用

```bash
sudo resize2fs /dev/vdb #我的设备名为vdb1
```

来完成扩容，同样还是使用`df -h /home`来检查结果。



### 1.3.2 CentOS虚拟机扩容

关闭虚拟机，然后再在宿主机(服务器)上找到虚拟机对应的映像文件。

利用`qemu-img`命令检查其大小并扩容，这个和Ubuntu一样

```bash
# 查看当前磁盘大小
sudo qemu-img info /home/public/virt-manager/dailybuild/centosvmdisk-dailybuild.qcow2
# 扩容
sudo qemu-img resize /home/public/virt-manager/dailybuild/centosvmdisk-dailybuild.qcow2 +100G
```

然后启动虚拟机，查看磁盘大小是否出现变更：
```bash
# 在虚拟机中查看磁盘大小
sudo fdisk -l /dev/* # 一般是vdb

# 查看当前分区和挂载情况
lsblk

df -h
```

一般情况下我们扩容的都是挂载在`/home`目录下的磁盘。

有可能出现该磁盘并未分区，是直接挂载的裸设备，这样的话扩容操作更加简单。

**未分区的磁盘**

首先检查该磁盘的文件系统

```bash
sudo file -s /dev/vdb
lsblk -f
```

根据文件系统类型进行扩容：

* 文件系统为EXT4(或者EXT2/EXT3)

  ```bash
  sudo resize2fs /dev/vdb
  ```

* 如果系统是XFS：

  ```bash
  sudo xfs_growfs /home
  ```

  与`resize2fs`直接操作设备不同，`xfs_growfs`命令操作的是文件系统的**挂载点**。

最后运行命令检查是否扩容成功：
```bash
df -h /home
```

**有分区的磁盘**

1. 首先运行分区工具：

   ```bash
   sudo fdisk /dev/vdb
   ```

2. 查看并记录分区信息：输入`p`打印分区表。**特别注意第一个分区的起始扇区**！(Start栏的值)，例如`2048`。后续操作这个很关键！

3. 删除旧的分区：输入`d`删除现有的分区(如果只有一个分区，则会自动选中)。删除分区不会擦除数据
4. 创建新分区：输入`n`创建新分区；选择主分区(`p`)或逻辑分区，一般为`p`；分区号保持与原来的一致，比如`1`；**第一个扇区**：必须输入我们刚刚记录的其实山区的值！者才能保证数据不丢失！**最后一个扇区**：直接按回车使用默认的最大可用扇区。
5. 写入并退出：输入`w`回车，将新的分区表写入磁盘

然后重启虚拟机`sudo reboot`。

此时完成了虚拟机磁盘的内部扩容，虚拟机能够识别到扩容后的磁盘，但是此时文件系统还不能识别到。

接下来就可以按照上面的未分区的操作来，根据不同的文件系统来执行命令就行了。

# 2 为虚拟机加入动态内存管理(弃置)



## 2.1 什么是动态内存管理

在实际操作之前，我认为有必要先整体了解一下动态内存管理的概念以及其实现方法。

### 2.1.1 动态内存管理（DMM）的核心目标

简单说，动态内存管理的目的是：**让有限的物理内存 “按需分配” 给多个虚拟机，提高整体利用率**。

- 当虚拟机需要内存时（如运行大型程序），宿主机 “临时扩容”。

- 当虚拟机空闲时（如程序关闭），宿主机 “回收内存” 给其他虚拟机。

  最终实现：“1 台宿主机的 16G 内存，能支撑多个虚拟机总需求 20G 的场景”（通过动态调度）。

#### 内存超分配（Overcommit）

宿主机允许为所有虚拟机分配的 “最大内存总和” 超过自身物理内存（例如宿主机 16G，给 3 个虚拟机各分配 8G，总和 24G）。

- 这是动态管理的前提（否则固定分配，无需动态调整）。
- 但超分配有风险：若所有虚拟机同时 “提现”（使用最大内存），会导致宿主机内存不足，需配合其他机制缓解。

#### 气球驱动（Balloon Driver）：内存 “搬运工”

这是动态调整的**核心执行组件**，作用是在虚拟机内部 “搬运” 内存：

- **膨胀（Inflate）**：当宿主机需要回收内存时，气球驱动在虚拟机内 “充气”，占用虚拟机的空闲内存，然后将这部分内存 “还给” 宿主机。
- **收缩（Deflate）**：当虚拟机需要更多内存时，气球驱动 “放气”，释放占用的内存，供虚拟机内的程序使用。

形象比喻：气球就像一个 “内存缓冲区”，通过大小变化在宿主机和虚拟机之间 “转移” 内存。

#### 通信机制：宿主机与虚拟机的 “对话”

气球驱动的膨胀 / 收缩需要 “指令”，这依赖双向通信：

- **虚拟机→宿主机**：通过`qemu-guest-agent`报告内部内存压力（如 “我现在需要更多内存”）。
- **宿主机→虚拟机**：通过 libvirt 发送内存调整指令（如 “请收缩气球，释放 2G 内存”）。

### 2.1.2 宿主机与虚拟机的必备组件（完整依赖链）

要让动态内存管理工作，宿主机和虚拟机需要各自安装对应的 “工具” 和 “驱动”，缺一不可：

#### 宿主机必备组件

1. **虚拟化层（KVM/QEMU）**
   - 作用：提供虚拟机运行的基础环境，支持内存的动态分配。
   - 依赖：`qemu-kvm` 包（提供虚拟化核心功能）。
2. **管理工具（libvirt）**
   - 作用：管理虚拟机的内存配置（最大内存、当前内存），触发动态调整。
   - 关键命令：`virsh setmem`（手动调整）、`virsh dommemstat`（监控）。
   - 依赖：`libvirt-daemon` 服务（后台运行，处理内存调度）。
3. **内存监控与调度模块**
   - 作用：检测宿主机内存压力，决定是否需要回收虚拟机内存。
   - 依赖：内核自带的内存管理模块（如`ksm`内存合并，减少重复内存占用）。

#### 虚拟机必备组件

1. **气球驱动（virtio-balloon）**
   - 作用：执行内存的膨胀 / 收缩（核心执行组件）。
   - 验证：虚拟机内执行 `lsmod | grep virtio_balloon`，有输出说明已加载。也有可能直接被编入内核，没有单独的模块。则通过`grep CONFIG_VIRTIO_BALLOON /boot/config-$(uname -r)`来查看，显示为`y`说明被编入内核。
   - 依赖：虚拟机内核支持`virtio`框架（几乎所有现代 Linux 内核都默认支持，需在虚拟机 XML 中配置`<memballoon model='virtio'/>`）。
2. **通信代理（qemu-guest-agent）**
   - 作用：在宿主机和虚拟机之间传递信息（内存压力、调整指令）。
   - 验证：虚拟机内执行 `systemctl status qemu-guest-agent`，显示`active (running)`。
   - 依赖：虚拟机内安装`qemu-guest-agent`包，且宿主机虚拟机 XML 中配置了通信通道（`<channel type='unix'>`）。
3. **内核内存管理配置**
   - 作用：允许气球驱动访问和调整虚拟机内存。
   - 关键内核配置（通常默认开启）：`CONFIG_VIRTIO_BALLOON`、`CONFIG_VIRTIO_PCI`。

### 2.1.3 组件交互流程（动态调整的完整步骤）

以 “虚拟机需要更多内存” 为例，各组件协作流程：

1. 虚拟机内程序申请内存，导致内存压力上升（`free -h`显示可用内存不足）。
2. `qemu-guest-agent`检测到压力，将信息通过通信通道发送给宿主机的`libvirt`。
3. `libvirt`检查宿主机是否有空闲内存，若有则向虚拟机发送 “收缩气球” 指令。
4. 虚拟机内的`virtio-balloon`驱动收到指令，释放自身占用的内存（气球收缩）。
5. 虚拟机内的程序获得更多内存，`virsh dommemstat`显示`actual`值增加。



### 2.1.4 构建完整动态内存管理的 checklist

- **宿主机**：安装`qemu-kvm`、`libvirt`，启动`libvirtd`服务，虚拟机 XML 配置正确（最大内存、气球模型）。
- **虚拟机**：加载`virtio-balloon`驱动，安装并启动`qemu-guest-agent`，内核支持`virtio`。
- **核心关联**：宿主机与虚拟机通过`qemu-guest-agent`通信，通过`virtio-balloon`执行内存调整。



## 2.2 修改虚拟机.xml文件

首先是需要修改对应虚拟机的xml配置文件。先关闭对应虚拟机，然后在主机终端中查找所有虚拟机：

```bash
virsh list - -all
```

查看对应虚拟机的xml配置

```bash
virsh dumpxml <虚拟机名称>
```

安全编辑虚拟机xml文件

```bash
virsh edit <虚拟机名称>
```

查找在`<device>`节点内部是否有`<memballoon>`这个元素
类似的配置会是：

```xml
<device>
    ...(其他配置，比如<disk>，<interface>等)
    <memballoon model=’virtio’/>
<device>
```

或者更详细一点：

```xml
<device>
    ...
    <memballoon model=’virtio’>
        <address type=’pci’ domain=’0x0000’ bus=’0x00’ slot=’0x07’ function=’0x0’/>
    </memballoon>
<device>
```

如果存在上述代码，则已经存在`Balloonn`设备，模型是`virtio`，我们现在需要在此基础上添加释放功能。
将其修改为：

```xml
<device>
		...(其他配置，比如<disk>，<interface>等)
		<memballoon model=’virtio’ autodeflate=’on’/>
	<device>
```

或者

```xml
<device>
	...
    <memballoon model=’virtio’ autodeflate=’on’>
	    <address type=’pci’ domain=’0x0000’ bus=’0x00’ slot=’0x07’ function=’0x0’/>
    </memballoon>
<device>
```

一般编辑是采用`vim`，保存退出后会显示：

```bash
Domain <虚拟机名称> XML configuration edited.
```

这表示修改成功，符合XML语法。

## 2.3 Virt-Manager GUI操作



### 2.3.1 QEMU-GUEST-AGENT

注意到基本上现在使用的虚拟机内核都编入了Balloon驱动，并且Virt-manager也支持在线修改当前分配的内存，只是缺少动态调整能力。

需要额外在虚拟机处安装qemu-guest-agent服务，该服务是宿主机和虚拟机沟通的桥梁。未安装该服务的虚拟机无法感知到宿主机为其分配的最大内存，因此也无法在内存不够时去申请更多内存。

所以需要安装该服务来实现动态内存调整。



尝试了很多，走了很多弯路...
下载软件包，查询`balloon`模块位置，启用服务等等...
最后发现`virt-manager4.0.0`以及目前使用的虚拟机版本都支持直接在图形界面中调整内存大小。还可以在首选项中打开监控CPU，内存等的占用情况。

![004](004.png) 
![005](005.png) 

然后为每个虚拟机的硬件-内存中配置当前内存大小和最大内存大小。

![006](006.png) 

为每个虚拟机分配2GB当前内存，作为内核运行的最小内存， 然后根据具体的需要按需为每个虚拟机分配最大内存。保证使用过程中动态获取和释放内存。
简单来说虚拟机拥有了 2GB静态内存和6GB动态内存（8GB最大内存为例）。

如果虚拟机开启了Balloon机制和qemu-guest-agent服务，并且最大内存大于当前分配内存，那么动态内存机制就会启动。
